Section 1
背景与整体流程图
5.0 的核心是收敛主链路，而不是继续叠加概念

为什么要从 4.0 整理成 5.0

  • 重复太多：Trigger / Action / Memory 在多个章节反复出现。
  • 对象混乱：群助手 / Personal Assistant / Flow Agent 同时被写成产品角色和系统模块。
  • 权限与等级交叉过深：容易误读成 “Level 越高 = 越能自动执行”。
  • Planning 定位不稳：有时像中间态，有时像独立卡片体系。

5.0 的统一规则

  • 前台永远只有一个用户可感知 AI：Personal AI Assistant
  • 短动作只走：Trigger -> Vote -> Execute
  • 复杂动作走：Trigger -> Planning -> Vote -> Execute
  • 长期任务升级成 Flow，文件留存在 Git Cloud Drive

5.0 最终主链路：Signal Detection → Trigger Chip → Planning → Vote → Execute → AI Secretary Review → Memory Update → Memory Dashboard

Section 3
Trigger 的设计与交互逻辑
Trigger 是最小可行动信号，而不是又一张复杂卡片

定义

Trigger 是 AI 从连续协作行为中抽取出的“现在值得处理”的最小决策单元。

设计原则

只表达一件事；不承载复杂执行；可追溯；在群聊内保持轻量不打断。

交互去向

群聊里点击进入右侧 Assistant；私聊里可直接进入 Vote；Flow 中进入项目事件流。

Section 4
Planning 的设计与交互逻辑
Planning 是复杂任务的执行准备态，不是独立产品物种

何时进入 Planning

  • 需要调用 2 个及以上外部工具
  • 需要读取或生成文件
  • 需要多步执行且存在依赖
  • 需要跨系统校验数据
  • 预计执行时长超过 10 秒

Planning 的产物

  • Step List：步骤拆解
  • Inputs：上下文来源
  • Risks：当前识别到的风险
  • Draft：待审批草稿
  • Tool Scope：将调用的工具和权限

Planning 交互逻辑

场景交互方式
简单任务不显示 Planning，直接进入 Vote
多步任务先显示 Planning 摘要，再提交 Vote
长耗时任务Planning 持续更新步骤状态
用户要求修改回到 Planning 重算，生成新 Vote
🧭 Planning Preview
准备生成客户会后跟进
3 steps · Gmail + CRM + Calendar
1. 拉取会议纪要和参会人
2. 生成 follow-up 邮件草稿
3. 准备 CRM note 和 next step 更新
风险预检：收件人有 1 位外部联系人；会议纪要里包含金额信息。
Section 5
Action Card（Vote）的设计与交互逻辑
前台统一只保留 Vote 心智，复杂度差异通过内容深度表达

Vote 的统一结构

  • Header：动作标题 + 目标对象
  • Context：信号来源和上下文摘要
  • Draft / Payload：将发送或更新的具体内容
  • Options：A / B / Decline
  • Permission：涉及工具和授权范围

Vote 状态机

draft -> proposed -> user_editing -> approved -> reviewing -> executing -> done / failed / blocked / deferred

5.0 关键收敛：Permission Vote 是 Vote 的一种，不再另起一套 UI 心智。

Section 6
群聊里的信号收集
AI 不在群里“发言”，而是在群里“观察、理解、触发”

采集对象

  • 会议意图
  • 任务分配
  • 决策形成
  • 明确同意 / 拒绝 / 待定
  • deadline、owner、依赖关系
  • 需要某个角色回应的问题

核心模块

  • Message Parser
  • Thread Aggregator
  • Participant Resolver
  • Cross-channel Resolver
  • Threshold Engine

5.0 定义：群聊确认收集 是系统能力，不是单独用例；群聊 Context 模式 继续保留，但本质上是 Personal AI 读取当前群语境，而不是一个独立群机器人在工作。

Section 7
Role Awareness 与 Git Cloud Drive
角色感知决定“给谁看什么”，Git Cloud Drive 负责“把结果记下来”

Role-aware 输出规则

角色默认输出
发起人 / Owner完整 Vote + 风险 + 执行按钮
Manager摘要 + 审批选项 + 升级建议
Contributor与自己相关的任务或回复建议
Watcher低打扰摘要，不默认推执行

Role Awareness 的输入

  • 组织角色：title / team / manager 关系
  • 项目角色：owner / approver / watcher / contributor
  • 历史行为：经常审批什么、忽略什么
  • 当前上下文：正在看的群、当前 Flow、最近会议
  • 权限范围：能看什么、能批什么、能执行什么

Git Cloud Drive 的职责

  • 所有正式交付物都文件化
  • 所有文件都有版本，不覆盖历史
  • 文件和 Trigger / Vote / Flow / 执行记录互相链接
  • 文件既是产出物，也是长期记忆源
Section 8
智能权限管理与 AI 秘书兜底二次审核提醒
Level 影响默认确认深度，但不会突破高风险安全底线

确认级别

Level名称行为
L1Silent自动执行，时间线可追溯
L2Notify执行后通知，可短时撤回
L3Confirm标准 Vote 审批
L4Deep Review增强上下文审核
L5Multi-gate多步逐级确认

权限模型

  • Tool Permission：工具能不能用
  • Action Permission：动作能不能做
  • Confirmation Level：做到什么确认深度
权限 Vote 统一三选项
  • 仅这次
  • 在当前项目 / Flow 内始终允许
  • 拒绝

AI Secretary Review

  • 收件人 / 对象是否正确
  • 时间与时区是否合理
  • 数据是否一致
  • 附件是否完整
  • 是否重复执行
  • 是否越权

兜底提醒机制：用户点击通过后，并不总是立即落地执行。若执行前复核发现问题，系统回到用户面前提示 “等一下，我发现一个风险”，并给出继续执行、修改后执行、取消三个后续动作。

Section 9
Memory Dashboard
从“设置页”升级为 AI 和用户之间的协作透明层

新的等级设计

Level名称定义默认状态
Lv.0未建立未连接工具、无稳定记忆、只能被动聊天初始默认
Lv.1观察中已接入基础数据，能识别低风险信号新用户
Lv.2试用中可做轻量提醒和草稿生成完成首批连接
Lv.3新同事能稳定完成标准协作任务已有持续审批记录
Lv.4得力助手对常规工作形成稳定默契低风险动作高通过
Lv.5左右手在低风险、高频任务上高度自治长期稳定使用

等级定义初始化和升级逻辑表格

Level进入条件升级条件降级条件
Lv.0默认初始连接至少 1 个核心工具并产生首个可用 Trigger无
Lv.1有基础连接7 天内产生有效 Trigger 且用户处理 ≥ 3 次长期无使用
Lv.2已有轻量互动14 天内批准 ≥ 5 次，成功执行 ≥ 3 次通过率持续过低
Lv.3有稳定协作30 天内多源信号稳定、拒绝率低、至少 1 个项目持续跟进错误率或拒绝率明显升高
Lv.4已形成工作默契高频低风险动作具备稳定模式，用户开始开放项目级权限风险拦截频繁、权限被回收
Lv.5长期稳定协作60 天以上稳定使用，多个 Flow 持续运行，文件化记忆和角色感知完整高风险误判或长期停用

更多的数据维度看板

维度说明
Productivity产出量：完成了多少动作和交付物
Approval Quality通过率：批准、拒绝、修改分布
Coverage覆盖了多少群、项目、联系人、工作域

Automation DepthL1-L5 动作分布
Permission Footprint连接了哪些 app、开放到什么粒度
Role Fit不同角色触发和采用情况
File Memory产出物数、版本数、被回用次数
Live Status正在监控、规划、等待审批、执行中
Section 10
文件化记忆留存
所有对未来有复用价值的产出，都必须文件化，而不只是留在聊天里

必须文件化的内容

  • 会议纪要
  • 决策记录
  • 周报 / 月报 / brief
  • 项目阶段总结
  • 客户沟通摘要
  • 研究型长文档
  • Flow 的中间和最终交付物

文件化记忆统一结构

  • memory_file_id
  • title
  • type
  • linked_signal_ids
  • linked_vote_ids
  • linked_flow_id
  • version
  • status
  • owners
  • tags

沉淀逻辑：Trigger / Flow -> Planning -> Draft File -> Vote -> Approved File -> Git Cloud Drive -> Dashboard & Retrieval
