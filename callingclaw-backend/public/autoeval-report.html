📋 Experiment Background

CallingClaw is a real-time voice AI that joins Google Meet as a participant, speaks, presents slides, and controls the computer. This report tracks the quality of its voice-driven presentation pipeline through automated experiments following the autoresearch pattern: each experiment has a clear goal, measurable metric, and keep-or-discard decision.

Core challenge: Coordinate three systems in real-time — OpenAI Realtime voice model (System 1), Claude Haiku intent classifier + action executor (System 2), and Playwright-controlled Chrome browser — to deliver coherent meeting presentations where AI speaks, scrolls, clicks, and responds to user questions without interrupting itself or losing context.

Methodology: Voice commands sent via sendText to simulate user speech. Each step verified across 3 layers: system health (Chrome alive, voice connected), tool accuracy (correct tool called), and content quality (specific info, no filler). Results persisted to autoeval-results.tsv for trend tracking.

💡 Key Learnings
Primer message — gpt-realtime-1.5 needs explicit behavioral framing at session start. Without it, model halluccinates for first 2 turns. With it, all 6 sections correctly use injected content.
response.create = dangerous — Sending response.create cancels in-progress audio. Direction A: stop ALL background response.create. Only user speech (VAD) and presentSlide() should trigger responses.
response.done ≠ audio done — OpenAI's response.done fires BEFORE audio.done. Flushing on response.done causes next response to cancel still-playing audio.
conversation.item.create = safe — Silent context injection. Model sees it on next turn. No audio interruption. All background updates should use this, not response.create.
Ring buffer 10s → 30s — Playback worklet overflowed on long responses (>10s). Increased to 30s.
Prompt ROI diminishes — 6 rounds of prompt tweaking (80→82→71→76→67→66%). Better to fix injection mechanism than polish prompt words.
