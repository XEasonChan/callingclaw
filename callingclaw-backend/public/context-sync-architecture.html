全景概览 — 双系统认知架构 ARCHITECTURE

CallingClaw 采用双系统认知架构 (Dual-Process)，模仿人类的快思考 (System 1) 和慢思考 (System 2)。五类 AI 角色通过 EventBus + 文件系统 + WebSocket 三层同步机制协同工作。

同步机制 — 五层传输 SYNC

A. EventBus (核心 Pub/Sub)

文件: src/modules/event-bus.ts

三路分发:

  • 本地监听: on(type, callback) 进程内订阅
  • WebSocket: ws://4000/ws/events 推送到 Desktop UI
  • Webhooks: HTTP POST + HMAC 签名, glob 模式匹配

History: 最近 200 事件 (可查询)

Correlation: startCorrelation("mtg") → 全会议追踪

interface CallingClawEvent {
  type: string;              // e.g. "meeting.started"
  timestamp: number;
  correlationId?: string;   // "mtg_1234567890ab_xyz"
  data: Record<string, any>;
}
B. Voice Session Update (全量替换)

方法: OpenAI Realtime session.update()

触发源:

  • ContextSync refresh (每 60s)
  • MeetingPrepBrief injection (会议开始)
  • OpenClaw liveNotes (via callback)
  • Computer Use task completion
  • ContextRetriever results

入口: voice-persona.ts::pushContextUpdate()

!
session.update 是全量替换 instructions + tools, 不是 patch。liveNotes 累积导致 token 线性增长。
C. 文件系统同步 (SharedDocuments)

文件: src/modules/shared-documents.ts

目录: ~/.callingclaw/shared/

  • {meetingId}_prep.md — OpenClaw 写入 (会前)
  • {meetingId}_live.md — CallingClaw 追加 (会中)
  • {meetingId}_summary.md — CallingClaw 写入 (会后)
  • {meetingId}_transcript.md — 完整记录
  • sessions.json — 会议索引 (最近 50 个)

meetingId 格式: cc_{timestamp}_{random}

D. IPC Bridge (Electron ↔ Backend)

文件: callingclaw-desktop/src/preload/index.js

暴露 API: window.callingclaw.*

  • daemon: start, stop, restart, status, onStatus, onError, onLog
  • permissions: check, request, openSettings
  • env: check (Bun, Python, BlackHole, etc.)
  • overlay: show, hide
  • skill: check, install
  • shell: openExternal
  • app: info

全部通过 ipcRenderer.invoke() (Promise-based)

E. Python Bridge (Audio + Input)

文件: src/bridge.ts

传输: WebSocket on port 4001

已知 Gap 与优化方向 GAPS

Gap 1: session.update 全量替换

OpenAI Realtime 的 session.update 是全量替换 instructions, 不支持增量 patch。

  • liveNotes 累积导致每次 push token 线性增长
  • 30 分钟会议后 system prompt 可达 ~9.5K chars

方向: liveNotes 分层 (pinned + recent + archived) + 周期压缩

Gap 2: Auditor 单一 Chunk 策略

TranscriptAuditor 给 Haiku 发送 15 条 unclassified entries, 包含大量噪声。

  • 不区分 command / requirement / decision / chat
  • 浪费 tokens 且降低准确率

方向: EntryClassifier (零成本规则引擎) 前置分类 → micro-task 拆分

Gap 3: System 2 会中被动

OpenClaw 在会议中只接收 batch push, 缺少周期性主动参与。

  • 无 "每 60s" 的 checkpoint 评估
  • 无法及时检测 topic 转换或需求变化

方向: PeriodicCheckpoint (60s / 20 entries) 状态评估 + 压缩

Gap 4: 后台任务单线程

TranscriptAuditor 有 single processing lock + 5s cooldown, 无法并行处理。

  • 不同类型任务共享同一个执行通道
  • "打开文件" 和 "记录需求" 无法同时进行

方向: MicroTaskScheduler — 4 个独立队列, 各自 debounce/cooldown

Gap 5: Context Retriever 与 Auditor 解耦

ContextRetriever 和 TranscriptAuditor 独立运行, 但共享 transcript 输入, 缺少协同。

  • Retriever 的 topic 信息未被 Auditor 利用
  • 可能重复分析相同内容

方向: 共享 topic state, Retriever 为 Auditor 提供 context enrichment

Strength: Topic Cache 316x 加速

ContextRetriever 的 topic-based cache 已经实现显著优化。

  • 86% cache hit rate (conversation turns 5-9)
  • 110ms → 0.35ms per lookup
  • Last 5 topics retained (LRU)

方向: 可扩展到 Auditor 使用相同 cache, 进一步减少 Haiku 调用

Eng Review 决策记录 2026-03-19

以下决策在 /plan-eng-review 中确认，作为优化路线图的实施约束。

# 决策 选项 Rationale
1 Topic 检测: 单一权威 ContextRetriever 是唯一 topic 源, 移除 TopicTracker micro-task DRY — 避免 3 个系统独立检测同一件事。Retriever 已有 topic cache (316x)。
2 EntryClassifier: Gemini Flash nano-classifier 用 Gemini 3 Flash (~30 tok/entry) 替代 regex 规则引擎 Regex 无法处理中英混合 (code-switching)。Gemini Flash 比 Haiku 便宜 15x、快 200ms。
3 liveNotes 压缩: 双层安全网 压缩推给 Voice, 但完整原文保留在 live.md + SharedContext recall_context 可从完整记录回查, 零信息丢失。压缩只影响 Voice system prompt token。
4 Dead config: 修复并引用 Vision 改用 CONFIG.screen.ssimThreshold, 值改为实际的 0.7 Config 声明 0.95 但代码用 0.7 — 消除误导, 让 threshold 可配置。
5 Topic state 共享 Retriever 暴露 getCurrentTopic(), micro-tasks 附带 topic context Auditor 知道 topic 后分类更准确 — "帮我打开那个" + topic="SSO" → 推断 SSO PRD。
6 测试策略: Full eval suite 50+ 双语 entries 做 eval dataset, 每个 micro-task >85% accuracy baseline LLM 集成不能 mock — 需要真实双语 transcript 验证分类准确率。
7 API 调用量: 微批量 累积 3-5 entries 后一次性分类, 调用量与当前持平 (~15 次/会议) 避免 OpenRouter rate limit, 延迟增加 ~3-5s 可接受。
