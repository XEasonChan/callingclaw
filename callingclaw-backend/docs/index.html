Section 02

问题分析 Pain Points

当前架构在长会议场景下的五个核心瓶颈。

Problem #1 — 无界增长

Transcript 无界膨胀

每次 Haiku 调用发送 15 条 transcript (~3000 tokens)。30 分钟会议产生约 450 条，触发约 30 次调用。

~450 entries / 30min → 30 Haiku calls
Problem #2 — 视觉延迟

屏幕描述时效性差

VisionModule 每 8 秒截图，batch 5 后约 40 秒才处理一次。上下文到达 OpenClaw 时已经过时。

~40s 延迟 per batch
Problem #3 — 全量替换

Voice Brief 全量重建

每次 ContextSync 都会 session.update 全量替换 brief。即使只有 1 行变更，也要重新推送 4K~8K tokens。

全量 session.update = 浪费
Problem #4 — 无优先级

所有内容同等权重

日常寒暄（"嗯好的"）和关键决策（"确认用 Electron 架构"）被同等处理，无法区分优先级。

P1 决策 == P5 闲聊
Problem #5 — Token 成本

成本线性增长不可控

30min 会议约 30 次 Haiku 调用，每次 ~$0.001。会议越长成本越高且无上限。

30 calls × $0.001 ≈ $0.03 / 30min
Section 03

提案 — 增量 Delta 架构

Section 04

Delta 类型定义

六种 delta 事件类型，按优先级从 P1 到 P5 排列，高优先级事件优先进入 accumulator。

Type Priority Source Trigger Token Est. 示例 Example
decision P1 Transcript 关键词检测 ~50 tokens "确认用 Electron 架构"
requirement P2 Transcript 关键词检测 ~80 tokens "需要支持 OAuth scope 预览"
action_result P2 EventBus automation.done ~30 tokens "[DONE] PRD opened in VS Code"
emotion_shift P3 Transcript 情绪变化 ~20 tokens "用户语气变焦虑"
utterance P4 Transcript 每条用户发言 ~40 tokens "我们看看下一个议题"
screen_observation P5 VisionModule 8s 截图 ~60 tokens "Screen: PRD permission section"
Section 05

后台任务定义 Background Tasks

BackgroundTaskScheduler 管理的 5 个异步任务，全部使用 Haiku 以最低成本运行。

Task 触发条件 Trigger Model Token 预算 Output
context_summarize 每 5min 或累计 2000 tokens Haiku 200 tokens out 上下文摘要，替换旧摘要段落
intent_classify 用户发言后 debounce 2.5s Haiku 100 tokens out 意图标签 + 置信度分数
requirement_extract 每 2min Haiku 150 tokens out 结构化需求条目列表
decision_track 收到 decision delta 时 Haiku 100 tokens out 决策记录 + 关联上下文
follow_up_detect 每 3min Haiku 100 tokens out 待办事项 + 负责人

Section 07

实现路径 Implementation Roadmap

分三个阶段渐进式交付，每个阶段独立可用，逐步替换当前管线。

v2.5.0

Phase 1 — Delta 基础设施

  • 实现 ContextDeltaStream 事件总线（EventEmitter 模式）
  • 实现 DeltaAccumulator + 优先级队列（500 token / 30s 窗口）
  • 定义 6 种 delta type 的 TypeScript interface
  • 在 TranscriptAuditor 中添加 keyword-based decision/requirement 检测
  • unit tests + integration tests with mock transcript stream
v2.6.0

Phase 2 — 后台任务 + 增量更新

  • 实现 BackgroundTaskScheduler（30s tick，async queue）
  • 实现 5 个后台任务（summarize, classify, extract, track, detect）
  • 实现 IncrementalBriefUpdater（diff-patch，section-level update）
  • 替换当前 ContextSync 的全量 session.update 逻辑
  • A/B 测试框架：同时运行 current + proposed，比较 token 消耗
v2.7.0

Phase 3 — 长会议优化 (Temporal Graph RAG)

  • 对超过 1 小时的会议引入 Temporal Graph 结构
  • 基于时间线的 RAG 检索，按相关性拉取历史上下文
  • 自动分段（topic segmentation）+ 段落级别摘要
  • 支持跨会议上下文关联（cross-session context linking）
  • 实现 context window 的动态伸缩（adaptive budget allocation）
