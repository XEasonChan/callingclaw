session.update 的关键限制

⚠️
OpenAI Realtime 的 session.update 是全量替换（不是 patch）——每次 push 必须重建完整 instructions + tools。随着 liveNotes 累积，每次 push 的 token 开销线性增长。当前没有"只推增量"的能力。

核心问题 PROBLEM

问题 1: Context 窗口 Chunk 太大

TranscriptAuditor 每次给 Haiku 发送 15 条 transcript entries（~500-700 input tokens），不区分内容类型。

  • 15 条中可能 10 条是闲聊，2 条是 automation 指令
  • Haiku 需要从噪声中找信号 → 浪费 tokens + 降低准确率
  • 没有把 context 按"信息类型"拆分成小块
问题 2: 没有增量任务定义

当前所有任务只有一种"通用 audit"：判断用户是否在发指令。

  • 缺少：需求捕捉、决策记录、action item 提取、topic 切换检测
  • 这些任务各自需要不同大小的 context chunk
  • 应该定义不同的任务类型，而不是一个万能 audit
问题 3: System 2 无周期性检查点

OpenClaw (System 2) 目前只在会前/会后参与，会议中被动等 batch push。

  • 缺少 "每 N 条 / 每 M 秒" 的周期性 checkpoint
  • 无法及时发现 topic 转换、需求变化
  • 应该有固定间隔的 bit-sized 任务重新评估状态
问题 4: Background Tool Call 不够细粒度

Voice AI 的 tool call 是顺序阻塞的，TranscriptAuditor 有 5s cooldown。

  • 无法同时处理 "打开文件" + "记录需求" + "更新 brief"
  • 需要将后台任务拆成独立的微任务队列
  • 让不同类型的任务有独立的执行通道

增量 Context Sync 方案 PROPOSED

核心思路：把一个大的"通用 audit"拆成多个小的、有类型的 micro-task，每个任务只消耗它需要的 context chunk。引入周期性 checkpoint让 System 2 定期评估会议状态。

触发: EntryClassifier 检测到 command 类型

Context Chunk:

  • 最近 3-5 条 command entries
  • Meeting brief 的 filePaths[] + browserUrls[]
  • 最近 5 条 [DONE] actions (去重参考)

执行: confidence ≥ 0.85 → AutomationRouter

回推: [DONE] task: result → Voice liveNotes

T2: QuickRecall (实时)

触发: Voice AI 的 recall_context tool call

T3: RequirementCapture (近实时)

Context Chunk:

  • 当前 topic 摘要
  • 已记录的 requirements (避免重复)

T4: DecisionRecord (近实时)

触发: 检测到 decision entries 或 15s debounce

Context Chunk:

  • 最近 3-5 条 decision entries
  • 相关的 requirements (提供背景)
  • 已记录的 decisions (避免重复)

Haiku 输出: { decision, rationale, relatedReqs[] }

回推: [DECISION] text → Voice + batch → MEMORY.md

T5: TopicTracker (周期)

触发: 每 20-30 秒，或检测到明显话题切换

Context Chunk:

  • 最近 2 条 entries
  • 当前 topic + keyPoints (from brief)
  • 极小输入: ~150 tokens

Haiku 输出: { changed: bool, newTopic?, keyPoints? }

回推: 更新 brief.topic → Voice system prompt 更新

T6: StateCheckpoint (周期)

触发: 每 60 秒或累积 20 条新 entries

Context Chunk:

  • 当前全部 liveNotes
  • 最近 10 条 entries 摘要
  • 当前 requirements + decisions 列表

执行: Haiku or OpenClaw

  • 压缩 liveNotes (合并重复、移除过时)
  • 生成状态摘要 (State Brief)
  • 建议下一步讨论方向

回推: 压缩后的 liveNotes → session.update (减少 token)

关键设计: Delta Push vs 全量 Push

session.update 优化策略

由于 OpenAI Realtime 的 session.update 是全量替换，我们无法真正只推"增量"。但可以通过以下策略减少每次 push 的 token 量：

策略 A: liveNotes 压缩 (StateCheckpoint)

  • 每 60 秒，T6 将 liveNotes 压缩：合并类似条目，移除已完成的 [DONE]，摘要化旧内容
  • 效果：liveNotes 维持在 ~15 条以内，不会无限增长

策略 B: 分层 liveNotes

liveNotes = {
  pinned: [          // 始终包含 (关键决策/需求)
    "[DECISION] 使用 OAuth2 SSO",
    "[REQ] 认证必须在 Q3 前完成",
  ],
  recent: [          // 最近 5 条 (滚动窗口)
    "[DONE] opened PRD in VS Code",
    "[SCREEN] 正在展示架构图",
  ],
  archived: "..."    // 压缩摘要 (只在 checkpoint 更新)
}

这样每次 session.update 只需要 pinned + recent + archived_summary，而不是全部历史 liveNotes。

策略 C: Push 频率控制

事件类型Push 策略原因
[DONE] command立即 push用户等待确认
[REQ] requirement批量 push (3 条或 15s)不紧急，减少 push 频率
[DECISION] decision批量 push (同上)同上
Topic 切换立即 pushVoice 需要知道当前在讨论什么
Checkpoint 压缩立即 push替换整个 liveNotes (减少 token)

并发安全设计

ℹ️
Bun 是单线程，所有微任务在同一个 event loop 中执行。并发安全通过以下方式保证：
  • 每个 Queue 有独立的 _processing lock（和当前 TranscriptAuditor 相同模式）
  • liveNotes 写入是顺序的（先 push 到 array → 再 session.update）
  • session.update 的 updateInstructions() 内部已经是 atomic 的
  • 不同 Queue 可以同时各自 debounce，但 Haiku API 调用在 await 时自动 yield

  • AUDITOR_MANAGED_TOOLS 不变 — 同样的 5 个工具在会议中移除
  • EventBus 事件保持兼容 (auditor.intent, auditor.executing, etc.)
  • 新增事件: auditor.requirement, auditor.decision, auditor.checkpoint
  • REST API 无变更 — 内部架构重构，对外透明

风险评估

风险 1: 本地规则的误分类

EntryClassifier 用正则规则分类，可能把命令误判为需求，或者需求漏判。

缓解: 规则保守 (宁可误判为 chat → 被 fallback 捕获)；Haiku 做最终判断；定期根据实际数据调优规则。

风险 2: liveNotes 压缩丢信息

StateCheckpoint 压缩 liveNotes 时可能丢失重要信息。

缓解: pinned 类型永不压缩；压缩前保存原始版本到 OpenClaw MEMORY.md；压缩规则可调。

风险 3: 多 Queue 竞争 Haiku API

多个微任务同时 debounce 到期，并发调用 Haiku。

缓解: Bun 单线程 + await → 自然排队；不同 debounce 时间天然错开；可加全局 rate limiter。
